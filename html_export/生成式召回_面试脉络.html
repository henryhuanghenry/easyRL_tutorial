生成式召回 · 面试脉络

📋一句话

生成式召回 (Generative Retrieval, GR) = 把召回从「embedding + ANN」改写成「自回归解码 item 标识符序列」，让 Transformer 端到端从用户历史直接生成推荐 item。代表演化主线：GENRE/DSI（思想前身）→ TIGER（Semantic ID 奠基）→ Codebook 改进百花（LETTER/LC-Rec/EAGER...）→ 工业落地（HSTU/OneRec/PinRec）→ 2026 新方向（并行解码/RL × GR）。

0. 面试 30 秒回答模板

面试官："你怎么理解生成式召回？"

给一个三段式回答：

  1. 是什么：把召回问题建模成"给定 user history，自回归生成 item 标识符序列"，跟 LLM 生成 token 一样，用 beam search 解码出 top-K item。
  2. 为什么：传统双塔召回受限于 embedding 表达力上限和 ANN 检索结构；生成式召回能借力 Transformer 的 scaling law、统一多任务、天然支持冷启动和多兴趣。
  3. 怎么做：(a) 用 RQ-VAE 把 item 量化成 3-4 token 的 Semantic ID（TIGER 范式），(b) encoder-decoder 自回归生成 next item 的 SID，(c) beam search + prefix trie 约束解码保证生成合法 item。
  4. 行业现状：Meta(HSTU)/快手(OneRec)/淘宝(RankGR)/腾讯微信(OneRanker) 都在工业落地，路线分化：HSTU 不用 SID 走 atomic ID + 巨大 embedding；快手走 SID + session-level beam；Pinterest 走 dense embedding + ANN。

1. 为什么会有生成式召回（动机/痛点）

1.1 传统双塔召回的痛点

痛点 表现
Embedding 表达力上限 双塔的 user/item embedding 是单个固定向量，复杂兴趣信息被压缩损失
多阶段 misalign 召回→粗排→精排→重排，每阶段优化目标不一致，cascade 误差累积
冷启动差 新 item 没有交互、embedding 训不出来；语义信息利用不足
Scaling law 弱 模型放大后 ANN 检索效率退化，工程上的 embedding table 也不能无限放大
多兴趣建模笨拙 单 embedding 难以表达多兴趣，要靠 multi-vector 或 capsule 等额外设计

1.2 生成式范式的吸引力

✨五大动机
  1. 借 LLM Scaling Law：模型/数据/算力的边际收益更高
  2. 端到端：召回→排序合并到一个生成模型里，避免级联误差（→ 召排一体）
  3. 多兴趣天然支持：beam search 出多条 SID 序列，每条对应一个兴趣
  4. 冷启动天然支持：Semantic ID 来自内容/语义，新 item 也能编码
  5. 可控/可解释：decoder 中间产出的 token 可以注入 ranking signal、做 reward-guided decoding

1.3 跟 LLM × 推荐 不是一个东西

概念 定义
LLM-as-Reranker 用 GPT/LLaMA 对候选集做 listwise 排序（RankGPT/Chat-REC）—— 不是生成式召回
LLM-as-Augmentor LLM 生成图增强信号、伪 label（LLMRec WSDM 2024）—— 也不是
LLM4Rec / TallRec / BIGRec 让 LLM 直接生成 item 标题，再做语义检索 → 介于两者之间
生成式召回（GR） 核心：item identifier 序列的自回归生成，TIGER 范式
AIGC 生成内容（GeneRec） 生成图文视频内容本身，扩充供给 → 狭义 vs 广义之分

面试官如果说"你了解 LLM 在推荐里的应用吗"，要明确区分以上 5 类，避免笼统作答。

2. 五阶段演化主线

阶段 1：思想前身（2021-2022）—— 自回归检索

论文 年份 一句话定位
GENRE (De Cao et al., ICLR 21) 2021 开山：BART encoder-decoder 逐 token 生成实体唯一名称，Constrained Beam Search + Prefix Trie 保证合法。内存仅 DPR 的 1/34
DSI (Tay et al., NeurIPS 22) 2022 把 GENRE 推广到文档检索，系统对比三种 docid（atomic / naive string / hierarchical k-means）
P5 (Geng et al., RecSys 22) 2022 把推荐统一成 "text-to-text" 多任务 prompt，T5 backbone，rating/sequential/explanation 等 5 类任务
ℹ️这阶段还不叫"生成式召回"

2021-2022 阶段，研究主体是 NLP 社区做的 generative retrieval，目标是文档检索（QA/Wikipedia entity linking）。但核心机制都已经具备：自回归生成 ID + beam search + prefix trie 约束。推荐圈正式接力是 TIGER (2023)。

阶段 2：Semantic ID 诞生（2023）—— TIGER 时刻

✨TIGER (Rajput et al., NeurIPS 2023, Google) —— 整个领域的奠基

核心 5 个组件（这是面试必背的 GR Q1-Q5 五问框架）：

  1. Codebook 构建：用 RQ-VAE 把 item 的 Sentence-T5 文本 embedding 量化成 4-token Semantic ID，每层 codebook size = 256
  2. 序列建模：encoder-decoder Transformer，输入 user 历史的 SID 序列，输出 next item 的 SID
  3. 训练目标：标准 next-token CE loss，teacher forcing
  4. Inference：标准 beam search，beam size = 20-50
  5. 评估：HR@K / NDCG@K，Amazon Beauty/Sports/Toys 三个标准 benchmark

TIGER 的影响：

  • 后续工作都基于 TIGER 改某一组件（codebook 改 RQ-VAE 的训练 / 模型架构换 LLaMA / inference 改 constrained / RL 后训练）
  • "RQ-VAE + 自回归生成"成为主流范式

阶段 3：Codebook 百花齐放（2024）

按改进维度分类，每类挑代表作：

改进维度 代表工作 关键创新

⚠️三个最容易被面试问的"经验法则"
  1. 协同信息 > 语义信息（来自 How-to-Index）：基于行为共现的 CID 比基于元数据的 SemID 效果更好
  2. 最优 SID 长度 3-4 token：太短表达力不够，太长破坏层级
  3. 语义主导问题（Semantic Dominance）：直接拼接语义+CF embedding 时，语义占 97%、CF 仅 3%，必须用 AdaLN/对比学习对齐才能融合

阶段 4：工业级落地（2024-2025）—— 路线开始分化

公司 系统 路线选择 线上效果 关键差异
Meta HSTU (ICML 24) 不用 SID，atomic ID + 10B embedding table，1.5T params ranking +12.4%, retrieval +6.2% pointwise gated attention（非标准 transformer），纯靠 scaling

快手广告 GR4AD SID + Beam Search + RSPO revenue ↑↑ RL 后训练

🔥工业落地的真坑（面试官最爱挖的）
  1. 推理延迟：自回归 beam search 解码占 91% 时间，必须做 speculative decoding (AtSpeed/GREAS) 或并行解码 (RPG/SETRec)
  2. Codebook 更新难：item 库每天新增百万，RQ-VAE 重训成本太高 → ETEGRec/PIT 的端到端动态更新
  3. 训练-推理不一致：teacher forcing vs autoregressive beam search → APAO/CARE/BEAR 类工作
  4. 冷启动 vs Codebook collapse：纯语义 SID 容易 collapse 到主流类目，纯 CF 又解决不了冷启 → 混合方案
  5. A/B 测试设计难：召回链路改 GR 后，下游精排目标分布飘移，要联合训练或大量 holdout 实验

3. 必背：4 个核心问题（面试官最爱问）

Q1：Codebook / Semantic ID 怎么构建？（RQ-VAE 详解）

RQ-VAE 算法步骤：

  1. 输入：item 的内容 embedding $\mathbf{x} \in \mathbb{R}^d$（如 Sentence-T5 输出）
  2. 第 1 层量化：$c_1 = \arg\min_k \|\mathbf{x} - e_k^{(1)}\|^2$，残差 $\mathbf{r}_1 = \mathbf{x} - e_{c_1}^{(1)}$
  3. 第 2 层量化：$c_2 = \arg\min_k \|\mathbf{r}_1 - e_k^{(2)}\|^2$，残差 $\mathbf{r}_2 = \mathbf{r}_1 - e_{c_2}^{(2)}$
  4. ...直到第 L 层
  5. 重构：$\hat{\mathbf{x}} = \sum_{l=1}^{L} e_{c_l}^{(l)}$
  6. Loss = MSE 重构 + commitment loss（让 $\mathbf{x}$ 接近选中的 codebook entry）+ codebook update（EMA 或 grad）

关键超参：

  • 层数 L：典型 3-4（TIGER 4×256、OneRec 3×8192、LETTER 4×256）
  • 每层 codebook size：256（学术）→ 8192/16384（工业）
  • 输入 embedding：Sentence-T5（学术主流）/ LLaMA / Qwen2.5-VL（多模态）

碰撞处理（同样的 SID 对应多个 item）：

  • TIGER：附加一个区分 token 作为第 5 位（去重）
  • 工业方案：直接保留碰撞，inference 时通过用户特征 disambiguate

RQ-VAE vs RQ-KMeans 对比：

维度 RQ-VAE RQ-KMeans
训练 端到端梯度 离线 K-Means
灵活性 可与下游联训（ETEGRec） 固定不可变
工业 主流 Spotify NEO 证明效果可能更好
稳定性 codebook collapse 风险 稳定

Q2：模型架构怎么选？

维度 encoder-decoder decoder-only
代表 TIGER, LC-Rec P5, BIGRec, HSTU
优势 历史/目标分离明确 与 LLM 兼容，迁移训好的 LLM 简单
劣势 参数量大一倍 长 context 拼一起

HSTU 为什么不用 SID 还能 work？

  • HSTU 的 attention 机制特殊：pointwise gated attention + position-aware bias
  • 直接学 atomic ID embedding，靠 1.5T params + 10B embedding table 的极致 scaling
  • 本质：HSTU 把 GR 范式视作"序列建模"问题，而非"标识符生成"问题
  • 训练比标准 transformer 提速 5.3-15.2x

Q3：Inference 怎么做？（beam search 类型谱系）

类型 代表 特点
标准 beam search TIGER, COST beam=20-50，invalid SID 直接丢弃（率 0.1-1.6%）
Constrained Beam GENRE, LC-Rec, SODA Prefix Trie / 非法 token 概率置 0
双流合并 EAGER, MQL4GRec 各流独立 beam, confidence-based 合并
Session-level Beam OneRec beam=128, 一次生成完整 5-item session
层次化 + RAG GLASS sid1 生成 → hard search → sid2/sid3
Speculative Decoding AtSpeed (ICLR 25) Draft model 加速，1.97x-2.11x
完全并行 SETRec, RPG OPQ 独立 token, 15x 加速
Multi-Token PinRec 每步生成 16 token, 10x 延迟降低
Dense + ANN PinRec, UniGRF AR 生成 dense embedding → Faiss ANN 检索（非 token 解码）

Q4：训练 vs 推理不一致（Exposure Bias）

问题：训练用 teacher forcing（每步喂 ground-truth 前缀），推理用 autoregressive beam search（每步喂自己生成的 token）—— 分布不匹配。

解决方案：

  • APAO（Yu 2026）：自适应前缀感知优化，训练时 sample 模型自己的 prefix
  • CARE：训练-推理对齐
  • Scheduled Sampling：经典 NLP trick，按概率混入模型自己的输出
  • BEAR（Yang 2026）：beam-search-aware regularization

5. RL × GR 衔接（跟你的 PPO/DPO/GRPO 笔记接上）

💡面试加分项：你能讲清"为什么 GR 必须做 RL 后训练"

SFT 阶段 GR 只学到了"用户曾经点过什么 → 接下来会点什么"的条件分布，没有内化"哪个 item 真正好（ranking signal）"。RL 后训练用 listwise/pairwise reward 修正这个偏差，让 SID 序列的 likelihood 跟 ranking signal 对齐——本质上和 RLHF 让 LLM 对齐人类偏好是同一回事。

7. 跟我的强项（RL）配套的回答模板

如果面试官问"你这有 RL 背景，能在推荐里怎么用"，可以这么答：

"RL 在推荐里有三个最佳切入点，正好对应你 GRPO/DPO/PPO 的三个角度：

第一，生成式召回的 RL 后训练——这是 2025-2026 最热的工业方向。SFT 阶段 GR 只学了 user history 的条件分布，但没对齐 ranking signal。OneRec 用 session-level GRPO 直接把 watch time 当 reward 优化整条 SID 序列；RankGR 用 listwise DPO 注入 ranking 偏好；GR4AD 用 RSPO 优化广告 revenue。

第二，多目标约束 RL——快手 TSCAC (WWW 23) 是经典案例，主目标 WatchTime + 辅助目标 Like/Share，用两阶段约束 Actor-Critic 求解。难点在密集信号 (WatchTime) 和稀疏信号 (Like) 共存时的方差控制，TSCAC 用 Multi-Critic 分离评估解决。

第三，Bandit / Contextual Bandit——召回-排序的 exploration-exploitation tradeoff，UCB/Thompson/LinUCB 是经典工具。"
